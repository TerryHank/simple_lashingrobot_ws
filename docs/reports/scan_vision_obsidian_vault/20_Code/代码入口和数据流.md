# 代码入口和数据流

## 入口 1：扫描流程

`run_pseudo_slam_scan(...)` 位于：

```text
src/tie_robot_process/src/suoquNode.cpp
```

固定识别位姿扫描会移动索驱到固定扫描位姿，然后调用：

```cpp
scan_srv.request.request_mode = kProcessImageModeScanOnly;
AI_client.call(scan_srv)
```

关键点：

- 当前固定扫描位姿只请求一次视觉。
- 如果视觉成功，就把响应里的点转换到 `map` 坐标系。
- 后续再做平面离群、棋盘格成员过滤和动态规划。

## 入口 2：视觉服务

`/pointAI/process_image` 位于：

```text
src/tie_robot_perception/src/tie_robot_perception/pointai/process_image_service.py
```

`request_mode=3` 会进入 `PROCESS_IMAGE_MODE_SCAN_ONLY`。

扫描模式拿到 `run_manual_workspace_s2_pipeline(...)` 输出后直接返回，不做多帧稳定：

```python
if request_mode == PROCESS_IMAGE_MODE_SCAN_ONLY:
    result = self.evaluate_point_coords_for_mode(latest_point_coords, request_mode)
    return result
```

## 入口 3：S2 / PR-FPRG

主入口位于：

```text
src/tie_robot_perception/src/tie_robot_perception/pointai/manual_workspace_s2.py
```

核心流程：

1. `prepare_manual_workspace_s2_inputs(...)`
2. `build_workspace_s2_rectified_geometry(...)`
3. `normalize_workspace_s2_response(...)`
4. `build_workspace_s2_axis_profile(...)`
5. `estimate_workspace_s2_period_and_phase(...)`
6. `build_workspace_s2_line_positions(...)`
7. `map_workspace_s2_rectified_points_to_image(...)`
8. `build_manual_workspace_s2_points_array(...)`

## 坐标转换

视觉输出的 `World_coord` 在扫描响应中仍是 `Scepter_depth_frame` 下的相机坐标。

流程层通过 TF 转成 `map` 坐标：

```text
Scepter_depth_frame -> map
```

然后写入：

```text
src/tie_robot_process/data/pseudo_slam_points.json
src/tie_robot_process/data/pseudo_slam_bind_path.json
```

## 当前薄弱边界

- S2 生成的是规则网格交点，不是逐点检测。
- 生成交点后只检查 mask 与附近三维坐标是否有效。
- 扫描模式没有多帧稳定。
- 规划层棋盘格过滤能保留「看起来规则」的假点，因此不能作为视觉质量兜底。
